# Web pool alignment and WalletConnect recovery — 2026-07-12

Status: **frontend code deployed and verified; Reown bare-origin allowlist still
requires the project owner's authenticated dashboard session.** No program,
IDL, SDK, backend, keeper, APK, economic, or on-chain files were deployed.

## Root causes

### Pool cards

Each card used an independent flex column with natural child heights. At the
four-column breakpoint, `No active round` and the left labels wrapped on only
some cards. That moved `.entry` by `19 px`. Premium's extra distinct-wallet
copy then made `.minimum-target` another `17.82 px` taller than the other
cards.

The card now has five explicit visual rows and uses CSS `subgrid` at multi-card
breakpoints. The label/name block reserves the same internal label row, the
minimum and facts rows stretch to the tallest peer, and the note/button live in
one footer row. Mobile remains natural-height and does not truncate Premium.

### WalletConnect

The old app loaded UniversalProvider and the modal from unpkg at runtime, then
awaited dependency loading, `UniversalProvider.init()`, and `display_uri`
without timeouts. Any stalled CDN/import/init/relay phase left the button in
`Opening WalletConnect…` forever.

The release now bundles the pinned UniversalProvider and local QR generator as
one same-origin ESM asset. It has separate timeouts for bundle loading (`8 s`),
provider initialization (`10 s`), and pairing URI (`12 s`). The user approval
wait starts only after `display_uri`, has no QR timeout, and always retains an
explicit Cancel path. Errors clear busy state and expose Try again. The loaded
module is cached across retries, double-clicks reuse one operation, stale
promises are rejection-handled, and session listeners are removed on cleanup.

The deprecated WalletConnect Web Component modal was removed. LuckyMe always
renders its own same-origin QR plus Copy URI, so retry cannot redefine custom
elements and no real URI is written into visible text, logs, screenshots, or
documentation.

## WalletConnect browser evidence

On `https://www.lucky-me.app`:

- same-origin bundle loaded;
- local QR and Copy URI appeared in about five seconds;
- Cancel removed QR/URI and re-enabled Connect;
- immediate retry produced a new QR;
- the final bundle emitted no warning/error on retry;
- no external `wcm-modal` remained in the DOM.

On `https://lucky-me.app`, Reown closed the relay WebSocket with code `3000` and
`Unauthorized: origin not allowed`. The UI reached its 12-second pairing-URI
timeout, showed a human error and Try again, and never remained infinite. A
second attempt behaved the same. `https://www.lucky-me.app` is authorized;
`https://lucky-me.app` must still be added to the Reown project allowlist from
an authenticated owner session at `dashboard.reown.com`.

Wallet Standard remains the first path for detected compatible extensions.
The direct mock test calls `standard:connect` and does not route Phantom,
Solflare, or Talisman-compatible Wallet Standard providers through
WalletConnect. The available in-app test browser had no installed wallet
extensions, so persistent-profile extension prompts still require a manual
Chrome/Brave owner smoke test.
